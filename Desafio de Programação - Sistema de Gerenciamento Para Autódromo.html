Desafio de Programação - Sistema de Gerenciamento Para Autódromo

Bem-vindo ao desafio de programação para a entrevista de emprego! Este projeto consiste em desenvolver um sistema full stack para um autódromo fictício.

Objetivo do Desafio

O objetivo deste desafio é avaliar suas habilidades na criação de um sistema distribuído, utilizando micro serviços, typescript, bancos de dados não relacionais e desenvolvimento de uma interface de usuário.

Requisitos Do sistema:

  1. Usuários: o sistema deve suportar 3 tipos de usuários.
    - Gestor
    - Afiliado
    - Comum
    Somente o usuário gestor pode cadastrar usuários, karts e pistas. O Usuário afiliado pode realizar agendamentos. Todos os usuários podem fazer uma corrida e acessar o histórico das mesmas.

  2. Pistas: O sistema deve permitir a criação de varias pistas. Para um usuário correr ou agendar corridas em uma pista ele deve ter seu cadastro vinculado a ela antes. As pistas devem ter nome, tamanho em km, quantidade de boxes, quantidade de lugares e endereço.

  3. Karts: os karts podem estar alocados em uma ou nenhuma pista. precisa ter um status que diga se ele está locado, se está em manutenção ou não está locado. Usuários gestores podem enviar e remover karts da manutenção. Os Karts devem ter nome, marca, modelo, potencia em cc (cilindradas), marca dos pneus e o status.

  4. Agendamentos: um agendamento reserva um kart especifico em uma pista especifica durante um período de tempo

  5. Eventos: Os eventos são informações temporais que devem ser armazenadas pelo sistema. Os eventos são classificados pelo seu tipo, podendo ser: Corridas, Manutenção e Acidentes. Os eventos serão criados e gerenciados via API. Não deve haver interface de usuário para criação deles.

  6. Corridas: As corridas são eventos criados via API, que possuem as seguintes informações: o usuário, o kart, a pista, o horário de inicio, horário de fim e se tinha agendamento. As corridas são criadas sem horário de fim e posteriormente são atualizadas com ele. Caso a corrida não possua horário de fim informado, o sistema deve interpretar que houve um acidente e a corrida foi abortada.

  7. Manutenção: São eventos criado via API que informam uma atividade de manutenção no Kart. Ele deve conter a informação de qual o kart, o motivo da manutenção, a data de entrada e a data de saída.

  8. Acidentes: Os acidentes são eventos que representam um acidente entre um ou mais corredores. Os acidentes devem conter os envolvidos, a pista e o horário. Acidentes dever ser criados via API.

  9. Notificação: quando uma corrida é iniciada ou um acidente acontece os usuários logados na plataforma com acesso a mesma pista onde houve o ocorrido devem receber uma notificação.

Da arquitetura:

  1. Arquitetura de Micro-serviços: O backend deve ser feito ser feito seguindo a seguinte arquiterura de microserviços, com cada serviço responsável por uma funcionalidade específica:

    - Autenticação
    - CRUD de entidades (User,Kart,Pista)
    - Agendamento de Kart
    - Eventos de corrida (Inicio, Fim, Acidente)
    - Eventos de Manutenção ( Envio do kart a manutenção. Retirada do kart da manutenção)

    A linguagem do backend deve ser, de preferência em nodejs com typescript

  2. Bancos de Dados Não Relacionais: Utilize bancos de dados não relacionais para armazenar as informações. Pode ser MongoDB, Cassandra, ou outra solução adequada.

  3. Frond-end: Desenvolva uma plataforma para a gestão. A plataforma deve permitir o cadastro de usuários, karts, pistas, agendamento de corridas, visualização de corridas anteriores e acidentes.
    Usuários gestores podem ver o histórico geral da pista;
    Usuários afiliados podem ver o histórico de acidentes geral porem apenas o seu histórico de corridas e;
    Usuários comuns apenas os acidentes e corridas que envolvam eles.
    A plataforma deve permitir que o usuário alterne entre as pistas que é cadastrado.
    De preferência em utilizar frameworks baseado em Typescript para criação do frontend.

 

  1. API Rest: Implemente uma API Rest para comunicação das camadas de frontend e backend, além da criação e atualização de eventos como corridas, acidentes e manutenção.

 

  1. Docker: Crie um Dockercompose que rode todo o sistema e exponha apenas a plataforma e a API
